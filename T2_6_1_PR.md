# T2.6.1: Expand role access for networking

**Intentional behavior change**, not a refactor. Follows T2.6 (route table refactor) by deliberately editing the permission matrix per product spec.

**Stats:** 2 files changed · +78 / −32 (net +46) · Build ✅ · Full lint ✅ (0 errors)

---

## The product decision

From the spec conversation:
> "All user types should have access to deal board, service requests, profiles so they can properly network, and the pipeline and analytics. Admins should only need to see the pages, not engage as a network member."

**Implementation:**
- **TC/Investor/PML** become full networking participants. Get the whole networking core: deal board, service requests, pipeline, analytics, plus all three profile-view routes.
- **Admin** becomes oversight-only. Gets admin/* routes + /service-requests for workflow observability. No deal-board, pipeline, analytics, own-profile, or profile-view routes. Admin doesn't engage as a network member.

## A side-effect bug fix

`InvestorCard.jsx` renders a "View Profile" link to `/profile/investor?id=<user_id>`, but that route **never existed in the route table**. Every user clicking that button was getting a 404.

Fixed as a natural consequence of adding `/profile/investor` to the member role access. Worth calling out so reviewers aren't surprised that this PR also fixed a broken link they may have noticed.

## What this PR does NOT do

**This PR grants route access only.** Admins can now navigate to `/service-requests` and see the page, but:

- The "New Service Request" button is still visible to admins
- The "Post a Deal" button on DealBoard is still visible to whoever can see the page
- The "Apply to this deal" flow is still available

Per your spec — "Admins should only need to see the pages, not engage as a network member" — these mutation affordances need to be hidden for admin. But that's **T2.6.2**, the follow-up PR. Splitting reasons:

1. **Cleaner review.** Access-grant and button-gating are different concerns.
2. **T2.6.1 unblocks the networking expansion immediately.** TCs, investors, and PMLs get their new access the moment this lands.
3. **Lower regression risk.** If hiding admin buttons accidentally breaks non-admin flows, we catch it isolated.

Between T2.6.1 landing and T2.6.2 landing, an admin clicking "Post Deal" would actually post a deal as admin. Not ideal, but also not dangerous — it's an entry, not a destructive action.

## Reviewer notes

- **`MEMBER_ROLES` constant** keeps the intent explicit. Anywhere in the code we want to say "everyone who is a network participant," we use `MEMBER_ROLES` instead of listing the three roles inline.
- **Sidebar is a second source of truth.** I considered driving it from the route table, but the nav has UI-specific fields (icons, labels, ordering) that aren't in the route entries. Added a header comment so the two stay in sync; a future refactor could fold the nav items into the route table if desired.
- **Admin's lost access to `/profile/tc` and `/profile/pml`** is deliberate. The pre-existing routes existed but admin had no real use for them — admin uses `/admin/members` for member oversight. Per the spec, admin doesn't engage as a network participant, so browsing to a specific TC's profile page wasn't appropriate anyway.
- **Why add /profile/investor now and not before?** The audit caught it as an "admin quirk" but missed that it was a broken link for every user. T2.6 preserved the quirk. T2.6.1 fixes it as part of the bigger access expansion.
