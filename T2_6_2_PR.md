# T2.6.2: Hide mutation affordances for admin (oversight-only)

Follow-up to T2.6.1. Per the product spec *"admins should only need to see the pages, not engage as a network member"*, hides the remaining mutation affordances that would let admin act as a transaction participant on pages they can now view.

**Stats:** 3 files changed · +58 / −43 (net +15) · Build ✅ · Full lint ✅ (0 errors)

---

## TL;DR: the scope shrank during reconnaissance

Going in, I estimated 5-8 component files would need changes. After auditing every mutation surface on every page admin gained access to in T2.6.1, most affordances turned out to be **already correctly gated** by role checks that happen to exclude admin.

**Already correctly gated (no change needed):**

| Page / component | Gate | Why admin doesn't see it |
|---|---|---|
| `DealBoard` — "Post a Deal" | `isInvestor` | admin != investor |
| `DealBoard` — "Edit Deal" on DealCard | `isInvestor` | admin != investor |
| `DealDetailModal` — "Apply" section | `userRole === 'tc'` | admin != tc |
| `RequestList` — "New Request" button | `currentUserRole !== 'admin'` | explicit admin check |
| Profile pages — "Edit Profile" button | `isOwner` | admin has no profile entity |
| Messages / SupportTickets | T2.6.1 route access | admin has no route |

**That left only three places** where admin could still act as a network participant. This PR fixes them.

---

## What changes

### 1. `src/lib/serviceRequestUtils.js` — `canAdvanceStatus` no longer includes admin

Before:

```js
export function canAdvanceStatus(userRole) {
  return userRole === "tc" || userRole === "admin";
}
```

After:

```js
/**
 * Can this user advance the status of a ServiceRequest?
 *
 * Only TCs can advance status — they are the workflow owners for the
 * transaction. Investors are read-only on the stepper (they wait for
 * the TC to move things forward). Admin is oversight-only (T2.6.2) —
 * admin can see the stepper but doesn't push transactions through it.
 * PMLs don't participate in the TC/investor ServiceRequest flow.
 */
export function canAdvanceStatus(userRole) {
  return userRole === "tc";
}
```

The "Mark as accepted / in_progress / completed" button is the single top-of-detail-pane action. Removing admin from this helper hides the button everywhere it's used. Investors were already read-only on the stepper; admin now joins them in that mode.

### 2. `src/components/service-requests/DocumentsSection.jsx` — hide Upload button for admin

Added a `canUpload` gate around the upload button:

```jsx
const canUpload = currentUser?.role && currentUser.role !== "admin";

return (
  <div>
    <div className="flex items-center justify-between mb-2">
      <h4 className="font-bold text-navy text-sm">Documents</h4>
      {canUpload && (
        <div>
          <input type="file" ref={fileInputRef} ... />
          <button onClick={...}>Upload</button>
        </div>
      )}
    </div>
    {/* Document list always renders — oversight read access */}
    ...
  </div>
);
```

Admin sees the document list for oversight but cannot add documents. The upload button is **hidden**, not just disabled — no affordance is rendered at all.

### 3. `src/components/service-requests/CommentsSection.jsx` — hide composer for admin

Same pattern with a `canPost` gate around the textarea + send button:

```jsx
const canPost = currentUser?.role && currentUser.role !== "admin";

// ...existing comment thread renders for everyone...

{canPost && (
  <div className="flex gap-2">
    <Textarea ... />
    <button onClick={handlePost} aria-label="Post comment">
      <Send className="w-4 h-4" />
    </button>
  </div>
)}
```

Admin reads the conversation for oversight. Cannot post new comments. Existing comments still render normally with the same styling as before.

## What's deliberately preserved

- **Admin still sees the full ServiceRequest detail pane.** Oversight requires visibility, not just bare route access. Admin sees:
  - Header with TC and Investor names
  - Progress stepper (animated, same as everyone else)
  - Timestamps grid (all four stage timestamps)
  - Notes section
  - Document list (read-only)
  - Comment thread (read-only)

- **Admin's engagement flows are untouched.** `/admin/applications`, `/admin/members`, `/admin/partners`, `/admin/support`, AdminDashboard's Approve/Reject actions, settings — all unchanged. Admin still engages where they're supposed to engage.

- **TC, Investor, PML UX is 100% unchanged.** They see the same buttons they've always seen, doing the same things they've always done. This PR adds three `canX` checks; none of those checks evaluate to false for non-admin roles.

- **T2.5 race fix is intact.** DocumentsSection and CommentsSection still perform refetch-before-write inside their handlers. The `canUpload` / `canPost` gates only wrap the UI; the mutation functions themselves are unchanged and still defensively check for `currentUser`.

## Reviewer notes

- **`canUpload` / `canPost` / `canAdvanceStatus` are the three consistent gates.** Naming them the same way makes it obvious where admin is restricted if you grep the codebase.
- **No handler-level guards added.** The thinking is: if admin doesn't see the button, they can't trigger the handler. Adding a handler-level `if (isAdmin) return` would be defense-in-depth but also adds noise. The existing handlers all check for `currentUser` before mutating; an admin who somehow invoked them (e.g., via dev console) would hit that same check and succeed, which is a minor issue but not a product-facing one. If stricter defense is desired, it's a one-line addition per handler.
- **Why hide vs. disable?** Disabling a button but leaving it visible would be confusing for admins — they'd wonder "why is this grayed out for me?" Hiding is clearer: the affordance simply doesn't exist in their view. Matches the spec's framing: "admins should only need to *see* the pages."
- **Pre-existing `currentUserRole !== 'admin'` in RequestList.jsx** (added during T2.4b as defensive coding) turned out to be load-bearing for this PR. Good foresight on that earlier commit; documented in T2.4b's PR body.
